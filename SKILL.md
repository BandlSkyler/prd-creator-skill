# PRD-TDD Creator - AI驱动文档生成 (v4.0)

面向程序员+AI协作的4阶段文档生成流程，产出可直接驱动代码生成的技术规格文档。

## 启动行为

当用户触发本技能时，AI 按以下逻辑判断入口：

1. **全新项目** — 从 Phase 1 开始，先向用户提问澄清需求。
2. **已有部分文档** — AI 扫描当前目录下 `prd/` 结构，识别已完成的阶段，从下一个未完成阶段继续。
3. **用户指定阶段** — 如用户明确说"只写 tdd"或"从架构拆分开始"，直接跳到对应阶段，缺少的前置信息通过提问补齐。

### 阶段判断逻辑

| 判断条件 | 入口阶段 | 说明 |
|----------|----------|------|
| 无 `prd/` 目录 | Phase 1 | 全新项目，从需求定义开始 |
| 有 `prd/` 但无 `P001/` 目录 | Phase 1 | 需求文档存在但未开始架构拆分 |
| 有 `prd/P001/README.md` 但无 `M001-*.md` | Phase 2 | 项目结构已创建，需完成模块文档 |
| 有模块文档但存在空 F 目录 | Phase 3 | 模块已定义，需填充功能规格 |
| 所有 F 目录有 prd.md + tdd.md | Phase 4 | 文档完整，进入自检验证 |

> **对话策略**: 每个阶段内，AI 先输出方案草稿，等用户确认或修改后再生成正式文档。不要一口气跑完所有阶段。

## 阶段总览

| 阶段 | 名称 | 核心动作 | 产出物 | 流转 |
|------|------|----------|--------|------|
| 1 | 需求定义 | 澄清需求 + 确认技术栈 | 需求文档 + 技术栈清单 | ⏸️ 用户确认 |
| 2 | 架构拆分 | 项目结构 + 模块依赖矩阵 | `P001/` 目录 + README + M文档 | ⏸️ 用户确认 |
| 3 | 功能规格 | 每个F目录产出 prd.md + tdd.md | 完整技术规格文档 | ⏸️ 每个F确认 |
| 4 | 自检验证 | AI执行检查清单 | 问题清单 + 修正方案 | ⏸️ 用户确认修正方案 |

> ⏸️ = 必须等用户确认后才进入下一阶段　🔄 = AI 自动执行

## Phase 1: 需求定义

> **流转**: ⏸️ 产出需求文档后，等用户确认再进入 Phase 2。

### 目标
一次性完成需求澄清和技术栈确认。

### 对话策略
1. **首轮**: 向用户提出 3-5 个关键问题（技术栈、核心场景、用户角色、约束条件），一次收集。
2. **追问**: 根据用户回答，针对模糊点追问 1-2 轮，不超过 3 轮对话。
3. **输出草稿**: 用模板填充需求文档草稿，请用户确认/修改。

### 必做步骤
1. **技术栈确认** - 收集后端/前端/数据库/缓存/MQ/部署方式
2. **需求澄清** - 用 5W2H 框架多轮对话（Who/What/Why/When/Where/How/How Much）
3. **功能优先级** - Must/Should/Could 三级划分
4. **范围界定** - 明确 In Scope / Out of Scope

### 产出
使用 `assets/requirement-template.md` 生成需求文档。

---

## Phase 2: 架构拆分

> **流转**: ⏸️ 产出项目 README + 模块清单后，等用户确认再进入 Phase 3。

### 目标
建立项目目录结构，完成模块划分和依赖分析。

### 拆分原则

| 层级 | 拆分依据 | 粒度 |
|------|----------|------|
| 项目 (P) | 独立部署、独立业务域 | 3-8个模块 |
| 模块 (M) | 单一职责、高内聚 | 3-8个功能 |
| 功能 (F) | 独立交付、可测试 | 一个用户场景 |

### 目录结构
```
prd/P{三位编号}-{项目名}/
├── README.md                    # 项目概述 + 技术栈 + M依赖矩阵
├── M001-{模块名}.md             # 模块文档（含F依赖关系）
├── M001/
│   └── F001-{功能名}/
│       ├── prd.md               # 业务需求（做什么）
│       └── tdd.md               # 技术规格（怎么做：DB+API+测试）
└── M002-{模块名}.md
```

### 路径规则

| 层级 | 格式 | 示例 |
|------|------|------|
| 项目 | `prd/P{三位}-{名}/` | `prd/P001-用户中心/` |
| 模块文档 | `Pxxx/M{三位}-{名}.md` | `P001/M001-用户认证.md` |
| 功能目录 | `Mxxx/F{三位}-{名}/` | `M001/F001-注册登录/` |
| 功能文档 | `Fxxx/{prd,tdd}.md` | `F001-注册登录/tdd.md` |

### 产出
- 项目 README（使用 Mermaid 画 M 依赖矩阵，并包含通用技术规范：请求头、统一响应格式、分页格式、错误码规范）
- 每个模块文档（使用 `assets/module-template.md`，含 F 依赖关系图）

### 模块依赖矩阵（放入 README.md）

| 模块 | 被谁依赖 | 依赖谁 | 开发顺序 |
|------|---------|--------|---------|
| M001-用户认证 | M002,M003 | 无 | 第1 |
| M002-账户管理 | M003 | M001 | 第2 |

## Phase 3: 功能规格

> **流转**: ⏸️ 每生成一个 F 目录的文档（prd.md + tdd.md），暂停等用户确认后再生成下一个。按模块依赖顺序、功能优先级顺序生成。

### 目标
为每个功能生成完整的技术规格文档，确保 AI 可直接基于文档生成代码。

### F目录文档

| 文件 | 必须 | 内容 | 模板 |
|------|------|------|------|
| prd.md | 是 | 业务需求、流程、规则、验收标准 | `assets/feature-template.md` |
| tdd.md | 是 | DB设计 + API定义 + 技术方案 + 测试用例 | `assets/tdd-template.md` |
| ui.md | 按需 | 有用户界面时生成 | `assets/ui-template.md` |

> **tdd.md 是技术规格的单一真相源**，包含数据库设计、接口定义、错误码、性能设计、测试用例，避免跨文件不一致。

### 建议生成顺序
```
prd.md（业务需求）→ tdd.md（技术规格：DB→API→测试）
```

### 文档间引用规范

**同功能内引用**（在 `prd.md` 中）:
```markdown
技术规格见 [tdd.md](./tdd.md)。
```

**同模块跨功能引用**（在 F002 中引用 F001）:
```markdown
依赖 [F001-注册/prd.md](../F001-注册/prd.md) 的用户注册能力。
```

**跨模块引用**（在 M002 的 F001 中引用 M001 的 F001）:
```markdown
依赖 [M001-用户认证/F001-登录/prd.md](../../M001-用户认证/F001-登录/prd.md) 的登录能力。
```

**在 README.md 中引用模块文档**:
```markdown
详见 [M001-用户认证.md](./M001-用户认证.md)。
```

**引用路径规则总结**:

| 引用场景 | 起始位置 | 目标位置 | 路径格式 |
|----------|----------|----------|----------|
| prd → tdd | F目录 | 同目录 | `./tdd.md` |
| F → F（同模块） | F目录 | 同模块其他F | `../F[编号]-[名]/prd.md` |
| F → F（跨模块） | F目录 | 其他模块的F | `../../M[编号]-[名]/F[编号]-[名]/prd.md` |
| README → M | 项目目录 | 模块文档 | `./M[编号]-[名].md` |

## Phase 4: 自检验证

> **流转**: ⏸️ AI 执行检查清单，输出问题清单后暂停，等用户确认修正方案后再执行修正。

### 目标
AI 对全套文档执行检查清单，发现问题后输出问题清单，由用户确认后再进行修正。

### 使用 `assets/review-checklist.md` 检查以下维度：

1. **完整性** - 每个F目录有 prd.md + tdd.md
2. **一致性** - prd 校验规则 ↔ tdd 中 API 字段校验 ↔ DB 字段定义
3. **可执行性** - 验收标准可测试、DDL可执行、接口定义无歧义
4. **依赖闭环** - F/M依赖关系完整、无循环依赖

### 执行流程

1. **执行检查** — AI 按 `assets/review-checklist.md` 逐项检查，输出问题清单
2. **暂停等待** — 将问题清单按风险等级分类展示，暂停等待用户确认
3. **用户确认** — 用户选择：
   - 确认修正方案 → AI 执行修正后重新检查
   - 跳过某些问题 → 标记为"已知风险"后继续
   - 手动修正 → 用户自行修改后，AI 重新检查
4. **循环直到通过** — 直到高风险问题为 0，中风险问题均有处理方案

### 检查通过标准
- 高风险问题：0个
- 中风险问题：已有处理方案
- tdd.md 中 DDL 可直接执行
- tdd.md 中接口定义无歧义
- tdd.md 中测试用例可转化为代码

## 从文档到代码

当文档套件准备好后，按以下顺序生成代码：

```
1. tdd.md (DB部分)  → 数据库 Migration 脚本
2. tdd.md (DB部分)  → 数据模型 (Model/Entity)
3. tdd.md (DB部分)  → Repository/DAO 层
4. tdd.md (API部分) → Controller/Handler 层
5. prd.md           → Service 层（业务规则）
6. tdd.md (测试部分) → 单元测试 + 集成测试
```

### 开发完成后验证
使用 `assets/dev-checklist.md` 逐项验证代码是否对齐文档规格：
- DB Migration ↔ tdd.md DDL
- API 实现 ↔ tdd.md 接口定义
- 业务逻辑 ↔ prd.md 规则
- 错误码 ↔ tdd.md 错误码表
- 测试用例 ↔ tdd.md 测试定义 + prd.md 验收标准
